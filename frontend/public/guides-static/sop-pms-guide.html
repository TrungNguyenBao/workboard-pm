Tổng quan hệ thống System Overview

Module PMS (Project Management System) cung cấp nền tảng quản lý dự án toàn diện — từ danh sách công việc đơn giản, bảng Kanban trực quan, đến Agile sprint với backlog và analytics chuyên sâu.

Mục đích
Theo dõi tiến độ dự án, phân công công việc, kiểm soát WIP và đo lường hiệu suất nhóm theo sprint.
Phạm vi áp dụng
Tất cả loại dự án: basic (danh sách), kanban (bảng kéo thả), agile (sprint + backlog + story points).
Tích hợp hệ thống
RBAC theo workspace/project, Real-time SSE, drag-drop (dnd-kit), biểu đồ burndown & velocity (recharts).
Nguyên tắc nền tảng
Mọi task được cập nhật trên hệ thống. Chỉ có 1 sprint active cùng lúc. WIP limit phải được tôn trọng.

The PMS (Project Management System) module provides a comprehensive project management platform — from simple task lists, visual Kanban boards, to Agile sprints with backlog and in-depth analytics.

Purpose
Track project progress, assign work, control WIP, and measure team performance by sprint.
Scope
All project types: basic (task list), kanban (drag-drop board), agile (sprints + backlog + story points).
System Integration
RBAC per workspace/project, real-time SSE, drag-drop (dnd-kit), burndown & velocity charts (recharts).
Core Principles
All tasks must be updated in the system. Only 1 active sprint at a time. WIP limits must be respected.

Agile SOP 06 — Quản lý Backlog & Ưu tiên hóa SOP 06 — Manage Backlog & Prioritization

Trang Backlog hiển thị theo kiểu Jira với sprint swim lanes. Mỗi sprint hiện thành một section có thể thu gọn, phần Backlog ở cuối chứa task chưa phân công sprint. Hỗ trợ kéo thả task giữa các section và context menu để di chuyển nhanh.

The Backlog page displays Jira-style sprint swim lanes. Each sprint appears as a collapsible section, with the Backlog section at the bottom containing unassigned tasks. Supports drag-drop between sections and context menu for quick moves.

Nguyên tắc Vàng Golden Principles

  • ★ Sự thật duy nhấtSingle Source of Truth — Mọi task phải tồn tại và được cập nhật trên hệ thống. Không trao đổi task qua chat mà không sync lên PMS.All tasks must exist and be updated in the system. Do not discuss tasks via chat without syncing back to PMS.
  • ★ WIP Limit là tín hiệu, không phải hình phạtWIP Limits Are Signals, Not Punishments — Cột đỏ nghĩa là bottleneck đang xảy ra — hãy dừng bắt đầu mới, hãy hoàn thành task hiện tại trước (Stop Starting, Start Finishing).A red column means a bottleneck is forming — stop starting new work, finish current tasks first (Stop Starting, Start Finishing).
  • ★ Sprint là cam kết, không phải dự báoSprints Are Commitments, Not Forecasts — Khi sprint đã active, không thêm task mới trừ khi có lý do khẩn cấp được Project Owner phê duyệt. Scope thay đổi liên tục phá hủy velocity.Once a sprint is active, do not add new tasks unless there is an urgent reason approved by the Project Owner. Constant scope changes destroy velocity.
  • ★ Estimate trước khi Sprint bắt đầuEstimate Before Sprint Starts — Tất cả task trong sprint phải có story points trước khi nhấn Bắt đầu Sprint. Analytics không có ý nghĩa nếu task thiếu estimation.All sprint tasks must have story points before clicking Start Sprint. Analytics are meaningless if tasks lack estimation.
  • ★ Kiểu dự án xác định quy trìnhProject Type Defines Process — Chọn đúng loại dự án ngay từ đầu: Basic cho nhóm nhỏ/đơn giản, Kanban cho luồng liên tục, Agile cho phát triển phần mềm theo sprint. Đổi loại sau có thể gây nhầm lẫn về dữ liệu.Choose the right project type from the start: Basic for small/simple teams, Kanban for continuous flow, Agile for sprint-based software development. Changing type later can cause data confusion.
